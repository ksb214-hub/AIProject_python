Crawling.py 설계문서 

1. 개요 (Overview)
    Crawling.py 모듈은 웹스크래핑을 통해서 레시피의 재료를
    추출하는 것을 목표로하는 웹 스크래핑 모듈이다. 
    사용자가 입력한 검색어를 기반으로 레시피의 제목, url를 추출하여 
    향후 상세 페이지로 이동하여 재료 및 레시피 데이터를 추출할 예정이다.


2. 역할 
    Crawling.py
        웹 페이지에서 레시피 제목과 url를 추출한다. 
        레시피 제목과 url를 바탕으로 향후 해당 레시피와 재료를 추출한다.

3. 사용기술
    사용 라이브러리 
        Python 
            Selenium
            bs4
            webdriver_manager
            time


4. 순서도 (개선)
    (1) 사용자/시스템으로부터 검색어 수집 (Batch 또는 On-demand)
    (2) driver_loader를 통한 독립적 브라우저 세션 생성
    (3) 검색 결과 스캔 및 common_sp_list_li 클래스 기반 요소 추출
    (4) [예외처리] 결과가 0개일 경우 '수집 불가' 로그 생성 후 종료
    (5) 상세 페이지 병렬 접속 (ThreadPoolExecutor)
    (6) 데이터 추출 및 DataCleaner를 통한 1차 정제
    (7) RecipeDataManager를 통해 리스트 -> 문자열 직렬화 및 DB 저장
    (8) driver.quit()을 통한 리소스 완전 해제`

6. 주요 Selector 정보
    Crawling.py는 다음 selector를 사용하여 데이터를 수집한다.

    레시피 개수 selector        
        #contents_area_full > ul > div > b
        검색 결과의 레시피 개수를 확인하는 selector이다.



        주의사항
            해당 selector는 DOM 구조에 의존하기 때문에 사이트 구조가 변경될 경우 동작하지 않을 수 있다.
            향후 class 기반 selector로 개선이 필요하다.



    레시피 목록 selector
        #contents_area_full > ul > ul > li
        검색 결과 페이지에 표시된 모든 레시피 카드 요소를 가져온다.



    재료 selector
        #divConfirmedMaterialArea > ul:nth-child(1) > li
        레시피 상세 페이지에서 재료 목록을 추출하는 selector이다.


시니어의 조언 (Technical Review)
    Selector의 유연성
        작성해주신 #contents_area_full > ul > ul > li는 구조가 조금만 바뀌어도 깨지기 쉽습니다.
        만개한레시피의 경우 레시피 카드가 공통적으로 common_sp_list_li 클래스를 가집니다. 
        추후 문제가 생기면 soup.select('.common_sp_list_li') 형태로 변경하는 것을 추천합니다.

    Anti-Bot 회피
        백그라운드(headless) 실행 시 서버에서 봇으로 감지하고 차단할 수 있습니다. 
        위 코드에 user-agent 설정을 넣어두었으니 차단을 최소화할 수 있을 겁니다.

    데이터 가공(re) 준비
        추후 상세 페이지에서 재료를 긁어올 때 100g, 2큰술, 약간 같은 텍스트가 섞여 들어올 텐데, 
        이때 정규표현식(re)을 사용하여 **[재료명, 수량, 단위]**로 파싱하는 로직을 별도 모듈로 분리하면 챗봇 답변 생성 시 훨씬 깔끔한 데이터를 줄 수 있습니다.

7. 추가사항 
    병렬 처리를 적용한 Crawling.py 구조 제안
        기존의 순차적(Sequential) 방식에서 병렬(Parallel) 방식으로 코드를 업그레이드

    멀티스레딩 주의 사항
        메모리 관리 (중요!)
            max_workers=4라면 크롬 브라우저가 동시에 4개 뜹니다. 
            맥북 프로 사양이라면 4~8개 정도가 적당합니다. 
            너무 많이 띄우면 오히려 시스템이 느려져서 다시 ReadTimeoutError가 날 수 있습니다.

        드라이버 독립성
            스레드들이 하나의 driver 객체를 공유하면 절대 안 됩니다. 
            각 스레드는 본인만의 driver를 생성하고(get_headless_driver), 
            작업이 끝나면 반드시 quit()으로 닫아줘야 메모리 누수를 막을 수 있습니다.

        서버 차단 방지
            너무 빠른 속도로 요청을 보내면 사이트 측에서 IP를 차단할 수 있습니다. 
            max_workers를 적절히 조절하고 필요하다면 중간에 랜덤하게 time.sleep을 섞어주는 센스가 필요합니다.

    에러의 원인: "데이터 타입 불일치"
        우리가 크롤링한 **재료(materials)**와 **조리 순서(steps)**는 파이썬에서 리스트(list) 형태입니다. 
        그런데 SQLite 같은 일반적인 데이터베이스(SQL)는 하나의 칸(Cell)에 '리스트'를 통째로 집어넣는 것을 허용하지 않습니다. 
        오직 문자열(String), 숫자(Integer/Float) 같은 기본 타입만 받을 수 있죠.
        즉, **"리스트를 문자열로 변환해서 저장해!"**라는 처리가 빠져서 발생하는 문제입니다
    
    해결 방법
        Crawling.py 내부의 데이터 가공 로직 수정
        _save_to_db 함수를 호출하기 전에, 리스트 형태인 데이터를 텍스트로 합쳐주는 과정(Serialization)을 추가해야 합니다.
